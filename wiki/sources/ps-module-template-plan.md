# PowerShell Module Template Repository — Plan

**Original**: `Projects/Powershell/PLAN.md`
**Date**: 2026-04
**Status**: Plan; companion to the already-built [[sources/pscodebase-scaffold]]

---

## TL;DR

Use a **GitHub Template Repository** (`PSModuleTemplate`) plus a local `bootstrap.ps1` initializer to spawn new PowerShell module repos. Branches, worktrees, forks, and Plaster/Catesta were all evaluated and rejected for the solo-operator profile. The template ships placeholder files (`{{ModuleName}}`, `{{ModuleGuid}}`, etc.); `bootstrap.ps1` replaces them, generates a fresh GUID, optionally adds the `BLUR.Ops` hub dependency for spoke modules, runs `git init` + initial commit, and optionally creates the GitHub remote. See [[concepts/scaffold-templating]].

---

## Strategy: GitHub Template Repo + Local Bootstrapper

| Strategy | Verdict | Why |
|---|---|---|
| Branches | Reject | Templates pollute history; merging template updates into projects is painful |
| Git worktrees | Reject | Tied to one repo; no independent history per project |
| Forks | Partial | Public-by-default; fork network visible; merging upstream is cherry-picking |
| Feature branches | Reject | A template isn't a feature |
| GitHub Template Repo | Recommended | One-click "Use this template" creates a fresh repo with no history link |
| **Template Repo + Bootstrapper** | **Best** | All of the above + automated placeholder replacement |

## How it works

1. `PSModuleTemplate` is marked **Template Repository** in GitHub Settings.
2. `tools/bootstrap.ps1` is included in the template — it prompts for `ModuleName`, `Author`, `Description`, `NounPrefix`, generates a fresh GUID, replaces all placeholders, optionally wires `BLUR.Ops` as a `RequiredModule`, and `git init`s.
3. New project workflow:
   - **Option A (GitHub UI)**: "Use this template" → clone → `./tools/bootstrap.ps1`
   - **Option B (Local)**: Clone template → delete `.git/` → `./tools/bootstrap.ps1`
   - **Option C (Claude Code)**: Read `PLAN.md` → scaffold directly
4. **No upstream sync.** Spawned projects diverge intentionally. Cross-cutting improvements use Claude Code to apply changes across repos.

## bootstrap.ps1 contract

```powershell
[CmdletBinding(SupportsShouldProcess)]
param(
    [Parameter(Mandatory)][ValidatePattern('^[A-Z][a-zA-Z0-9.]+$')][string]$ModuleName,
    [Parameter(Mandatory)][string]$Description,
    [string]$Author        = 'BlueRush',
    [string]$CompanyName   = 'BlueRush Inc',
    [string]$NounPrefix    = 'BLUR',
    [string]$MinPSVersion  = '7.2',
    [switch]$IsBlurOpsSpoke,
    [bool]$InitGit         = $true,
    [string]$GitHubRepo
)
```

Bootstrap actions:
1. Generate new GUID via `[guid]::NewGuid()`
2. Replace `{{ModuleName}}`, `{{ModuleGuid}}`, `{{Author}}`, `{{CompanyName}}`, `{{NounPrefix}}`, `{{MinPSVersion}}`, `{{Year}}`, `{{ModuleDescription}}`
3. Rename files containing `{{ModuleName}}` in their names
4. If `-IsBlurOpsSpoke`: add `RequiredModules = @('BLUR.Ops')` to manifest, `Import-Module BLUR.Ops -ErrorAction Stop` to top of `.psm1`
5. Self-delete `tools/bootstrap.ps1` (single-use)
6. Remove `.gitkeep`s from now-populated directories
7. If `-InitGit`: `git init`, `git add -A`, `git commit -m "feat: initial scaffold from PSModuleTemplate"`
8. If `-GitHubRepo`: `gh repo create $GitHubRepo --private --source=. --push`

## Files generated by the template

Same overall layout as the [[sources/pscodebase-scaffold]] but with placeholders. Notable additions over the scaffold:
- `tools/bootstrap.ps1` (the initializer itself)
- `Public/Get-{{NounPrefix}}Status.ps1` — example reference function with full `[CmdletBinding(SupportsShouldProcess)]`, comment-based help, error handling
- Module structural test verifying every public function has a matching test file
- `.github/workflows/ci.yml` and `publish.yml` per the [[sources/pscodebase-scaffold]] CI pattern

## BLUR.Ops Hub-and-Spoke

The `-IsBlurOpsSpoke` switch makes the template understand the BLUR.Ops ecosystem:

```
BLUR.Ops (hub)                       <- shared utilities, BLURResult class, config mgmt
├── BLUR.Ops.AWS         (spoke)
├── BLUR.Ops.IAM         (spoke)
├── BLUR.Ops.Network     (spoke)
├── BLUR.Ops.Security    (spoke)
└── BLUR.Ops.Compute     (spoke)
```

Each spoke declares `RequiredModules = @('BLUR.Ops')`. Bootstrap auto-wires this when invoked with `-IsBlurOpsSpoke`.

## Why Not Plaster/Catesta?

For a solo operator, `bootstrap.ps1` wins on:

- **Zero dependencies** — pure PowerShell
- **Fully transparent** — entire templating logic in one file
- **No XML manifests** — Plaster's `plasterManifest.xml` adds friction without payoff at this scale
- **Version-controlled alongside the template**
- **Runnable by Claude Code without installing anything**

If the team grows or templates start being published externally, migrate to **Catesta** (handles edge cases bootstrap.ps1 skips: encoding, conditional includes, user prompts).

---

## Implementation Sequence

1. **Phase 1 — Template repo**: Create `PSModuleTemplate/`, run the embedded prompt to generate files, validate (`Test-ModuleManifest`, `Invoke-ScriptAnalyzer`, `Invoke-Pester`), push as Template Repository, dry-run "Use this template" → bootstrap.
2. **Phase 2 — BLUR.Ops migration**: Compare existing BLUR.Ops scaffold against template, identify gaps, re-scaffold or selectively adopt.
3. **Phase 3 — Future projects**: Either GitHub UI or local clone + bootstrap.

## Template Versioning

Tag template repo with semver. `bootstrap.ps1` writes `# Scaffolded from PSModuleTemplate vX.Y.Z` into the generated `CLAUDE.md` so projects know their origin version.

---

## Other Design Decisions

- **Invoke-Build over psake** — task runs in script scope (cleaner isolation), better PS7 support
- **GitHub Actions over Jenkins for new projects** — OIDC federation removes long-lived AWS creds; native `shell: pwsh`. Jenkins remains supported for existing pipelines
